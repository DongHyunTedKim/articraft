# 카테고리 선정 기준 가이드 (Category Selection Requirements)

이 문서는 레포지토리의 `data/categories/` 하위에 위치하는 분류 체계(Taxonomy)에 새로운 카테고리를 추가하거나 기존 항목을 유지하기 위한 기준(Bar)을 정의합니다.

카테고리는 단순히 흥미로운 프롬프트 아이디어에 그쳐서는 안 됩니다. 
지속적으로 다량의 프롬프트 배치를 지원하고, 신뢰할 수 있는 여러 실제 레코드(records)를 담아낼 수 있어야 하며, 명확한 `target_sdk_version`을 지원하는 **영속적인 분류 체계 항목**이어야 합니다.

---

## 핵심 표준 기준 (Core Standard)

훌륭한 카테고리는 아래 요건들을 모두 만족해야 합니다:

- 명시적인 URDF 조인트(joints)를 가진, 신뢰 가능한 **관절 작동 객체(articulated object)**로 표현 가능할 것
- `sdk` 스택을 통해 안정적으로 빌드(buildable) 가능할 것
- 객체의 고유한 정체성(object identity)이 명확할 정도로 구체적일 것
- 일회성 설계를 넘어 다양한 변형 설계들을 지원할 수 있을 만큼 범용적일 것
- 기존에 이미 보유하고 있는 카테고리들과 명확히 차별화(distinct)될 것

만약 기획한 카테고리가 위 테스트 중 단 하나라도 통과하지 못한다면, 해당 분류 체계 단위는 적합하지 않을 가능성이 높습니다.

---

## 1. 관절 작동 적합성 (Articulation Fit)

카테고리는 본 레포지토리에서 지원하는 관절 작동 모델(articulation model)에 매끄럽게 매핑되어야 합니다:

- 명시적인 조인트로 연결된 **단단한 강체 부품(rigid parts)** 구조일 것
- `revolute`(회전 한계 있음), `continuous`(자유 회전), `prismatic`(병진/슬라이딩), 또는 `fixed`(고정) 모션 범주 내에 있을 것
- **트리 구조 운동학(tree-structured kinematics)**을 따를 것

본질적인 동작 매커니즘이 explicit 트리 조인트로 honestly 표현할 수 없는 아래의 기계 패턴들은 **단호하게 제외(Reject)**합니다:

- **폐루프 링크 구조 (Closed kinematic loops)**가 핵심 메커니즘을 이룰 때
- 케이블, 벨트, 또는 체인을 통한 회전/이동 전달 방식
- 도르래(pulley) 경로 라우팅
- 연동된 모션 구속 조건(motion constraints)이 필요한 기어 결합(gear coupling)
- 헬리컬 스크류(나사식 동력 전달) 운동이 주요 핵심 메커니즘일 때
- 래칫(ratcheting), 인덱싱(indexing), 또는 간헐적 물리 잠금이 주 모션일 때
- 정밀한 기하학적 시뮬레이션 트릭이 수반되어야만 움직이는 접촉 구동식 모션

### ⚠️ 위험하거나 모델링하기 까다로운 부적합 예시들:
- 폐루프가 설계의 본질인 가위형 링크 장치 (scissor jack, scissor lift 등)
- 래칫과 폴(ratchet-and-pawl) 장치
- 제네바 드라이브 (Geneva drives)
- 케이블-도르래 기반 리프트 (cable-and-pulley lifts)
- 기계식 차동 기어셋 (differentials)
- 물리적 접촉이 필수적인 캠-팔로워 시스템 (true cam-follower)

동작을 가짜(fake)로 흉내 내야만 구현할 수 있는 대상은 본 레포지토리의 카테고리로 적합하지 않습니다.

---

## 2. SDK 구현 적합성 (SDK Fit)

카테고리는 레포지토리가 이미 지원하는 도구와 지오메트리 패턴을 사용해 현실적으로 자동 구현(authoring)이 가능해야 합니다.

### 👍 구현하기 좋은 대상:
- 명확한 기계 프레임, 하우징, 쉘, 가이드 레일, 바퀴 허브, 브래킷, 포크, 아암(arms), 슬라이딩 트레이 또는 프레임 구조가 뚜렷할 것
- 지지 관계(parent/child support)가 명확한 단단한 하드서페이스 지오메트리
- 계통도가 읽기 편한 부모/자식 트리 구조
- 기본 입체 도형(primitives), 단면 압출 프로파일, 스윕(sweeps), 회전체(revolves), 불리언(booleans) 등의 정석적인 CadQuery 솔리드로 빌드할 수 있는 지오메트리

### 👎 구현하기 나쁜 대상:
- 연성 체(soft-body) 및 유연한 재질의 거동
- 천, 케이블, 인장 스프링, 체인 등 변형되거나 유연한 요소가 아이덴티티인 경우
- 극도로 유기적인 곡면 처리(highly organic freeform surfacing)
- 복잡하게 얽힌 다면 곡면 로프트 외관 바디나 조각상 형태의 외형 하우징
- 특수한 해킹이나 억지 결합 조건을 억지로 묶어주어야 작동하는 메커니즘

만약 프롬프트를 아주 정밀하게 작성했음에도 불구하고 컴파일, 품질 검사(QC) 또는 테스트 과정에서 에러를 자주 내뿜고 취약하다면, 이 레포지토리에 적합한 카테고리가 아닙니다.

## 3. 카테고리 범주 한계 (Category Scope)

하나의 카테고리는 너무 광범위한 우산형 분류도 아니어야 하고, 반대로 현미경 수준의 극단적인 파생형도 아니어야 하며, **안정적인 단일 객체 클래스 또는 단일 매커니즘 클래스**여야 합니다.

### 👍 올바른 범주 한계 (Good Scope):
- 하나의 일관된 객체 부류 또는 메커니즘 부류를 명확하게 지칭할 것
- 여러 예시들 간에 구조적인 일관성을 직관적으로 공유할 것
- 핵심 정체성을 잃지 않으면서도 다양한 프롬프트 변형(subtypes)을 지원할 것
- 상식선에서 예측 가능한 수준의 보편적인 관절 작동 세트를 가질 것

### 👎 잘못된 범주 한계 (Bad Scope):
- **너무 광범위함**: 서로 완전히 다른 프롬프트와 연결 구조 설계를 요구하는 여러 개의 객체군들을 억지로 묶어둔 경우
- **너무 좁음**: 사실상 하나의 고유한 기계 디자인 인스턴스에 카테고리 이름만 씌워 둔 경우
- **너무 모호함**: 생성 결과물의 내부 위상(topology)이나 지지대 결합 구조가 예측 불가능하게 매번 제멋대로인 경우

*만약 동일한 카테고리 안에서 '서로 다른 형태로 신뢰 가능한 여러 예시 모델'을 직관적으로 떠올릴 수 없다면, 그 범주 획정은 잘못된 것입니다.*

---

## 4. 기존 카테고리와의 독립성 (Distinctness)

새로운 카테고리는 분류 체계(Taxonomy) 상에서 스스로 그 존재 가치를 증명해야 합니다.
단순히 기존 카테고리의 이름만 바꾸거나, 미세한 변형에 그치는 수준이어서는 안 됩니다.

### ❌ 거절 대상이 되는 중복 카테고리 유형:
- 기존 카테고리에서 조인트 하나만 살짝 추가한 변형
- 기존 위상(topology) 구조에서 반복 부품 개수만 다르게 한 경우
- 기존 카테고리의 하위 서브셋(subset) 부류
- 기존 메커니즘에 단순 부착물만 추가한 버전
- 이미 존재하는 카테고리의 명칭만 바꾼 것
- 여러 기존 카테고리 영역과 심각하게 겹치는 모호한 상위 우산형 분류

### 👍 승인 대상이 되는 새로운 카테고리 조건:
- 의미 있게 완전히 차별화된 관절 구조 위상(articulation topology)을 도입할 때
- 완전히 다른 형태의 기계식 지지 프레임 구조(support structure)를 도입할 때
- 의미 있게 다른 작동 매커니즘 아키타입(archetype)을 도입할 때
- 객체 자체의 정체성과 형태가 완벽히 다를 때

## 📋 의사결정 최종 체크리스트

새로운 카테고리를 카탈로그에 등록하기 전에 아래 6가지 질문을 자문하세요:

1. [ ] 그 장치의 핵심 동작이 **명시적인 URDF 조인트**와 **순수 트리 운동학(tree kinematics)** 구조로 Honest하게 표현 가능한가?
2. [ ] `sdk` 빌드 엔진을 통해 컴파일 에러 없이 **다양하고 믿을 수 있는 형태의 예시들을 안전하게 빌드**할 수 있는가?
3. [ ] 카테고리의 경계 범주가 일관되고 안정적인가?
4. [ ] `data/categories/` 하위에 등록된 기존 카테고리들과 **물리적/개념적으로 완전히 구별**되는가?
5. [ ] 기하학 형상이 실제로 어떻게 설계되어야 하는지에 기반해 명확한 `target_sdk_version`을 지정할 수 있는가?
6. [ ] 이 카테고리가 단 하나의 단발성 럭키 프롬프트가 아니라, **수차례의 배치 구동과 수십 개의 통과 레코드를 지속적으로 지탱**할 수 있는 내구성을 가졌는가?

단 하나의 질문이라도 "아니오"가 나온다면, 해당 카테고리는 등록 보류되거나, 기존 카테고리로 통합되거나, 혹은 더 명확한 단위로 명칭 및 범위를 재정의해야 합니다.
